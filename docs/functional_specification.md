## 1. 공통 기능

### 1.1 로그인 / 인증 (AUTH)

| 항목 | 내용 |
|---|---|
| **ID** | CMN-AUTH-01 |
| **우선순위** | Must |
| **설명** | 관리자와 입주자 모두 **로그인 ID + 비밀번호**로 인증하며, 인증 성공 시 **JWT(Access Token + Refresh Token)** 을 발급받는다. |
| **역할 구분** | 로그인 시 역할(`ADMIN` / `RESIDENT`)을 판별하여 역할에 맞는 화면으로 리다이렉트한다. |

**상세 흐름:**

```
1. 사용자가 로그인 ID와 비밀번호를 입력한다.
2. BFF가 Spring 백엔드의 /api/auth/login 을 호출한다.
3. 백엔드는 DB에서 사용자를 조회하고, 비밀번호 해시를 검증한다.
4. 검증 성공 → JWT(Access + Refresh) 발급 → 응답.
5. 검증 실패 → 401 에러 + "아이디 또는 비밀번호를 확인하세요" 메시지.
6. BFF는 토큰을 httpOnly 쿠키에 저장하고, 역할에 따라 라우팅한다.
   - ADMIN  → /admin/dashboard
   - RESIDENT → /my-room
```

**유효성 검증:**

| 필드 | 규칙 |
|---|---|
| 로그인 ID | 필수, 4~50자 |
| 비밀번호 | 필수, 8자 이상 |

## 2. 입주자(Resident) 기능

### 2.1 내 방 기기 제어

#### 2.1.1 내 기기 목록 조회

| 항목 | 내용 |
|---|---|
| **ID** | RES-DEV-01 |
| **우선순위** | Must |
| **화면** | 입주자 앱 > 내 방 기기 제어 |
| **설명** | 로그인한 입주자의 **활성 계약(ACTIVE)** 에 연결된 Space에 설치된 기기 목록을 조회한다. |

**표시 정보:**

| 항목 | 설명 |
|---|---|
| 기기명 | 예: "천장 조명", "벽걸이 에어컨" |
| 기기 종류 | LIGHT / HVAC / LOCK / WASHER (아이콘으로 구분) |
| 현재 상태 | ONLINE (초록) / OFFLINE (회색) / ERROR (빨강) |
| 공간명 | 설치된 공간명 (예: "301호") |

**조회 로직:**

```
1. JWT에서 resident_id를 추출한다.
2. CONTRACT 테이블에서 해당 입주자의 ACTIVE 상태 계약을 조회한다.
3. 계약에 연결된 space_id로 DEVICE 테이블에서 is_active = true인 기기를 조회한다.
4. 기기 목록을 반환한다.
```

---

#### 2.1.2 기기 제어 (ON/OFF/설정)

| 항목 | 내용 |
|---|---|
| **ID** | RES-DEV-02 |
| **우선순위** | Must |
| **화면** | 입주자 앱 > 내 방 기기 제어 > 기기 상세 |
| **설명** | 기기 종류에 따라 적절한 제어 명령을 전송한다. |

**기기별 제어 명령:**

| 기기 종류 | 가능한 명령 | UI 형태 |
|---|---|---|
| LIGHT (조명) | `TURN_ON`, `TURN_OFF` | 토글 스위치 |
| HVAC (냉난방) | `TURN_ON`, `TURN_OFF`, `SET_TEMP` | 토글 + 온도 슬라이더 (16~30℃) |
| LOCK (도어락) | `LOCK`, `UNLOCK` | 토글 스위치 |
| WASHER (세탁기) | `START`, `STOP` | 시작/정지 버튼 |

**제어 흐름:**

```
1. 입주자가 기기를 선택하고 제어 버튼을 누른다.
2. 프론트 → BFF → Spring 백엔드로 제어 요청 전달.
3. 백엔드에서 권한 검증:
   - 해당 기기가 입주자의 활성 계약 Space에 속하는지 확인.
   - 기기 상태가 ONLINE인지 확인.
4. 권한 검증 통과 → 목업 IoT 서버로 명령 전송.
5. 목업 IoT 응답 수신 → CONTROL_LOG에 기록.
6. 결과를 프론트에 반환 (성공/실패 + 변경된 상태).
```

**에러 처리:**

| 상황 | 응답 | 사용자 메시지 |
|---|---|---|
| 권한 없음 | 403 | "해당 기기에 대한 제어 권한이 없습니다" |
| 기기 오프라인 | 409 | "기기가 오프라인 상태입니다. 잠시 후 다시 시도해주세요" |
| IoT 통신 실패 | 502 | "기기 통신에 실패했습니다" |

```
3. 검증 통과 → RESERVATION 레코드 생성 (status = PENDING).
4. 관리자가 승인하거나, 자동 승인 정책에 따라 APPROVED로 변경.
```

| **설명** | 전체 계약 목록을 조회한다. 상태별(ACTIVE/EXPIRED/TERMINATED), 입주자명, 호실별 필터링을 지원한다. |

**표시 정보:** 계약 ID, 입주자명, 호실명, 계약 기간, 월 임대료, 상태

#### 3.4.2 계약 등록

| 항목 | 내용 |
|---|---|
| **ID** | ADM-CTR-02 |
| **우선순위** | Must |
| **화면** | 관리자 대시보드 > 계약 관리 > 신규 등록 |
| **설명** | 입주자와 호실 간의 새로운 임대차 계약을 등록한다. |

**입력 항목:**

| 필드 | 필수 | 규칙 |
|---|---|---|
| 입주자 | ✅ | 기존 입주자 계정 중 선택 |
| 호실 (Space) | ✅ | PRIVATE 타입이며 AVAILABLE 상태인 공간만 선택 가능 |
| 계약 시작일 | ✅ | 날짜 형식 |
| 계약 종료일 | ✅ | 시작일 이후 |
| 월 임대료 | ✅ | 양수 (원) |
| 보증금 | | 양수 (원) |

**처리 로직:**

```
1. 해당 호실에 ACTIVE 계약이 이미 있는지 확인한다.
   - 있으면 → 409 "해당 호실에 이미 활성 계약이 존재합니다"
2. CONTRACT 레코드 생성 (status = ACTIVE).
3. 해당 Space의 status를 OCCUPIED로 변경한다.
```
