# CoLiving IoT 플랫폼 — 기능 명세서

> [!NOTE]
> 기획안(CoLiving-기획안.md)의 **섹션 4 범위 정의(MoSCoW)** 및 **섹션 5 기능 요구사항**을 기반으로,
> 각 기능을 **화면 단위**로 상세 기술한 문서입니다.
> 역할은 **일반 유저(User)**, **입주자(Resident)**, **관리자(Admin)** 세 가지가 존재합니다.

---

## 역할(Role) 정의

| 역할 | DB 값 | 설명 | 전환 조건 |
|---|---|---|---|
| **일반 유저(User)** | `USER` | 회원가입 시 기본 부여 역할. 방 조회, 예약 신청, 커뮤니티 열람이 가능하다. | 회원가입 시 자동 부여 |
| **입주자(Resident)** | `RESIDENT` | 계약 체결이 완료된 유저. 계약된 방의 IoT 기기 제어, 공용 시설 예약, 커뮤니티 참여가 가능하다. | 계약(CONTRACT) 생성 시 자동 승격 |
| **관리자(Admin)** | `ADMIN` | 시스템 전체를 관리하는 운영자. 모든 기능에 접근 가능하다. | 시스템에서 직접 부여 |

> [!IMPORTANT]
> **역할 승격 흐름:** `USER` → (계약 체결) → `RESIDENT` → (계약 만료/해지) → `USER`
> 계약이 만료되거나 해지되면 자동으로 `USER` 역할로 복귀하며, IoT 접근 권한이 회수된다.

## 1. 공통 기능

### 1.1 회원가입 (AUTH)

| 항목 | 내용 |
|---|---|
| **ID** | CMN-AUTH-00 |
| **우선순위** | Must |
| **설명** | 신규 사용자가 **로그인 ID, 비밀번호, 이름, 연락처**를 입력하여 회원가입한다. 가입 시 기본 역할은 `USER`로 설정된다. |

**입력 항목:**

| 필드 | 필수 | 규칙 |
|---|---|---|
| 로그인 ID | ✅ | 4~50자, 영문+숫자, 중복 불가 |
| 비밀번호 | ✅ | 8자 이상, 영문+숫자+특수문자 포함 |
| 비밀번호 확인 | ✅ | 비밀번호와 일치해야 함 |
| 이름 | ✅ | 2~50자 |
| 연락처 | ✅ | 전화번호 형식 |
| 이메일 | | 이메일 형식 |

**처리 흐름:**

```
1. 사용자가 회원가입 폼을 작성하여 제출한다.
2. BFF가 Spring 백엔드의 /api/auth/register 를 호출한다.
3. 백엔드에서 유효성 검증:
   - 로그인 ID 중복 확인.
   - 비밀번호 정책 확인.
4. 검증 통과 → USER 테이블에 레코드 생성 (role = USER).
5. 가입 완료 → 로그인 화면으로 리다이렉트.
```

### 1.2 로그인 / 인증 (AUTH)

| 항목 | 내용 |
|---|---|
| **ID** | CMN-AUTH-01 |
| **우선순위** | Must |
| **설명** | 모든 사용자(USER, RESIDENT, ADMIN)는 **로그인 ID + 비밀번호**로 인증하며, 인증 성공 시 **JWT(Access Token + Refresh Token)** 을 발급받는다. |
| **역할 구분** | 로그인 시 역할(`ADMIN` / `RESIDENT` / `USER`)을 판별하여 역할에 맞는 화면으로 리다이렉트한다. |

**JWT 토큰 Payload 구조:**

| 클레임(Claim) | 설명 |
|---|---|
| `sub` | 사용자 고유 ID |
| `role` | 역할 (`USER` / `RESIDENT` / `ADMIN`) |
| `contract_id` | 활성 계약 ID (RESIDENT만 포함, 없으면 null) |
| `space_id` | 계약된 공간 ID (RESIDENT만 포함, 없으면 null) |
| `exp` | 토큰 만료 시각 |
| `iat` | 토큰 발급 시각 |

> [!IMPORTANT]
> `RESIDENT` 역할의 JWT에는 `contract_id`와 `space_id`가 포함되어, 해당 입주자가 제어할 수 있는 IoT 기기의 범위를 토큰 레벨에서 제한합니다.

**상세 흐름:**

```
1. 사용자가 로그인 ID와 비밀번호를 입력한다.
2. BFF가 Spring 백엔드의 /api/auth/login 을 호출한다.
3. 백엔드는 DB에서 사용자를 조회하고, 비밀번호 해시를 검증한다.
4. 검증 성공 → 역할 확인:
   a. RESIDENT인 경우 → 활성 계약(CONTRACT) 조회 → contract_id, space_id를 토큰에 포함.
   b. USER/ADMIN인 경우 → 기본 클레임으로 토큰 발급.
5. JWT(Access + Refresh) 발급 → 응답.
6. 검증 실패 → 401 에러 + "아이디 또는 비밀번호를 확인하세요" 메시지.
7. BFF는 토큰을 httpOnly 쿠키에 저장하고, 역할에 따라 라우팅한다.
   - ADMIN    → /admin/dashboard
   - RESIDENT → /my-room
   - USER     → /rooms (방 목록/예약 화면)
```

**유효성 검증:**

| 필드 | 규칙 |
|---|---|
| 로그인 ID | 필수, 4~50자 |
| 비밀번호 | 필수, 8자 이상 |

### 1.7 비밀번호 변경 (PROFILE)

| 항목 | 내용 |
|---|---|
| **ID** | CMN-PRF-03 |
| **우선순위** | Must |
| **화면** | 앱 > 내 정보 > 비밀번호 변경 |
| **설명** | 현재 비밀번호를 확인한 후 새 비밀번호로 변경한다. |

**입력 항목:**

| 필드 | 필수 | 규칙 |
|---|---|---|
| 현재 비밀번호 | ✅ | 기존 비밀번호와 일치해야 함 |
| 새 비밀번호 | ✅ | 8자 이상, 영문+숫자+특수문자 포함, 현재 비밀번호와 다를 것 |
| 새 비밀번호 확인 | ✅ | 새 비밀번호와 일치해야 함 |

**처리 흐름:**

```
1. 사용자가 현재 비밀번호, 새 비밀번호, 새 비밀번호 확인을 입력한다.
2. 백엔드에서 현재 비밀번호 해시를 검증한다.
3. 검증 통과 → 새 비밀번호를 해시하여 DB에 업데이트한다.
4. 기존 Refresh Token을 무효화하여 재로그인을 유도한다.
5. 변경 완료 메시지를 표시하고 로그인 화면으로 리다이렉트한다.
```

### 2.3 예약(계약 신청)

| 항목 | 내용 |
|---|---|
| **ID** | USR-RSV-01 |
| **우선순위** | Must |
| **화면** | 유저 앱 > 방 상세 > 예약 신청 |
| **설명** | AVAILABLE 상태의 호실에 대해 입주 예약(계약 신청)을 제출한다. **로그인(USER 이상) 필수.** |

**입력 항목:**

| 필드 | 필수 | 규칙 |
|---|---|---|
| 호실 (Space) | ✅ | AVAILABLE 상태인 PRIVATE 공간만 선택 가능 |
| 희망 입주일 | ✅ | 오늘 이후 날짜 |
| 희망 계약 기간 | ✅ | 1개월 이상 |
| 연락처 | ✅ | 전화번호 형식 |
| 요청 사항 | | 자유 입력 (최대 500자) |

**처리 흐름:**

```
1. 유저(USER)가 방 상세 화면에서 "예약 신청" 버튼을 클릭한다.
2. 예약 신청 폼을 작성하여 제출한다.
3. 백엔드에서 검증:
   a. 해당 호실이 AVAILABLE 상태인지 확인.
   b. 해당 유저에게 이미 진행 중인 예약이 있는지 확인.
4. 검증 통과 → BOOKING 레코드 생성 (status = PENDING).
5. 관리자에게 예약 알림 전달.
6. 관리자가 승인 → BOOKING status = APPROVED, 계약 조건이 확정됨.
7. 유저에게 승인 알림 전달 → 유저가 계약 체결 화면에서 최종 동의.
```

> [!NOTE]
> 관리자 승인 후 즉시 계약이 생성되지 않습니다. 유저가 계약 조건을 확인하고 **동의(계약 체결)** 해야 최종 계약이 생성됩니다.
> 계약 체결은 [2.5 계약 체결](#25-계약-체결개인공간)을 참고하세요.

### 2.5 계약 체결(개인공간)

| 항목 | 내용 |
|---|---|
| **ID** | USR-CTR-01 |
| **우선순위** | Must |
| **화면** | 유저 앱 > 내 예약 > 계약 체결 |
| **설명** | 관리자가 승인(APPROVED)한 예약에 대해, 유저가 계약 조건을 확인하고 동의하여 최종 계약을 체결한다. 계약 체결 시 역할이 `USER` → `RESIDENT`로 승격된다. |

**계약 확인 화면 표시 정보:**

| 항목 | 설명 |
|---|---|
| 호실 | 계약 대상 호실명 (예: "301호") |
| 층 / 면적 | 호실 위치 및 면적 정보 |
| 계약 시작일 | 관리자가 확정한 입주 시작일 |
| 계약 종료일 | 관리자가 확정한 계약 종료일 |
| 월 임대료 | 관리자가 확정한 월 임대료 (원) |
| 보증금 | 관리자가 확정한 보증금 (원) |
| 특약 사항 | 관리자가 기재한 특약 사항 (있는 경우) |

**처리 흐름:**

```
1. 유저가 APPROVED 상태의 예약에서 "계약 체결" 버튼을 클릭한다.
2. 계약 조건 확인 화면이 표시된다 (임대료, 보증금, 기간 등).
3. 유저가 계약 조건을 확인하고 "동의 및 계약 체결" 버튼을 클릭한다.
4. 동의 체크박스 확인:
   - [필수] 계약 조건에 동의합니다.
   - [필수] 이용 약관에 동의합니다.
5. 백엔드에서 처리:
   a. 해당 예약이 여전히 APPROVED 상태인지 확인.
   b. 해당 호실이 아직 AVAILABLE 상태인지 확인.
   c. CONTRACT 레코드 생성 (status = ACTIVE).
   d. BOOKING status → CONTRACTED로 변경.
   e. 해당 Space status → OCCUPIED로 변경.
   f. 유저의 role → USER에서 RESIDENT로 변경.
6. 계약 완료 화면 표시 + 새 JWT 발급 안내.
7. 다음 로그인 또는 토큰 갱신 시 contract_id, space_id가 포함된 JWT가 발급된다.
```

> [!IMPORTANT]
> **역할 승격:** 계약 체결이 완료되면, 해당 유저의 역할이 `USER` → `RESIDENT`로 자동 변경되고,
> 다음 로그인 또는 토큰 갱신 시 `contract_id`와 `space_id`가 포함된 새 JWT가 발급됩니다.
> 이 시점부터 입주자는 계약된 방의 IoT 기기에 접근할 수 있습니다.

**에러 처리:**

| 상황 | 응답 | 사용자 메시지 |
|---|---|---|
| 예약이 APPROVED가 아님 | 409 | "이 예약은 현재 계약 체결이 불가능한 상태입니다" |
| 호실이 이미 OCCUPIED | 409 | "해당 호실이 이미 계약 완료되었습니다. 관리자에게 문의해주세요" |

## 3. 입주자(Resident) 기능

> [!NOTE]
> 입주자(RESIDENT)는 계약이 체결된 유저입니다. 아래 기능은 `RESIDENT` 역할을 가진 사용자만 접근 가능하며,
> JWT 토큰의 `space_id` 클레임을 기반으로 IoT 기기 접근 범위가 결정됩니다.

### 3.1 내 방 기기 제어

#### 3.1.1 내 기기 목록 조회

| 항목 | 내용 |
|---|---|
| **ID** | RES-DEV-01 |
| **우선순위** | Must |
| **화면** | 입주자 앱 > 내 방 기기 제어 |
| **설명** | 로그인한 입주자의 **활성 계약(ACTIVE)** 에 연결된 Space에 설치된 기기 목록을 조회한다. |
| **접근 제한** | `RESIDENT` 역할 필수. JWT의 `space_id` 클레임으로 접근 범위를 제한한다. |

**표시 정보:**

| 항목 | 설명 |
|---|---|
| 기기명 | 예: "천장 조명", "벽걸이 에어컨" |
| 기기 종류 | LIGHT / HVAC / LOCK / WASHER (아이콘으로 구분) |
| 현재 상태 | ONLINE (초록) / OFFLINE (회색) / ERROR (빨강) |
| 공간명 | 설치된 공간명 (예: "301호") |

**조회 로직:**

```
1. JWT에서 role을 확인한다. (RESIDENT가 아니면 403 반환)
2. JWT에서 space_id를 추출한다.
3. CONTRACT 테이블에서 해당 입주자의 ACTIVE 상태 계약을 검증한다.
4. JWT의 space_id와 계약의 space_id가 일치하는지 이중 검증한다.
5. 해당 space_id로 DEVICE 테이블에서 is_active = true인 기기를 조회한다.
6. 기기 목록을 반환한다.
```

#### 3.1.2 기기 제어 (ON/OFF/설정)

| 항목 | 내용 |
|---|---|
| **ID** | RES-DEV-02 |

| **설명** | 기기 종류에 따라 적절한 제어 명령을 전송한다. |

**기기별 제어 명령:**

| 기기 종류 | 가능한 명령 | UI 형태 |
|---|---|---|
| LIGHT (조명) | `TURN_ON`, `TURN_OFF` | 토글 스위치 |
| HVAC (냉난방) | `TURN_ON`, `TURN_OFF`, `SET_TEMP` | 토글 + 온도 슬라이더 (16~30℃) |
| LOCK (도어락) | `LOCK`, `UNLOCK` | 토글 스위치 |

**제어 흐름:**

```

3. 백엔드에서 권한 검증:
   a. JWT에서 role이 RESIDENT인지 확인.
   b. JWT의 space_id와 해당 기기의 설치 공간(space_id)이 일치하는지 확인.
   c. CONTRACT 테이블에서 해당 계약이 ACTIVE인지 확인.
   d. 기기 상태가 ONLINE인지 확인.
4. 권한 검증 통과 → 목업 IoT 서버로 명령 전송.
5. 목업 IoT 응답 수신 → CONTROL_LOG에 기록 (actor_type = RESIDENT).
6. 결과를 프론트에 반환 (성공/실패 + 변경된 상태).
```

**에러 처리:**

| 상황 | 응답 | 사용자 메시지 |
|---|---|---|
| 역할이 USER (입주자 아님) | 403 | "입주자만 기기를 제어할 수 있습니다. 계약을 먼저 체결해주세요" |

| 기기 오프라인 | 409 | "기기가 오프라인 상태입니다. 잠시 후 다시 시도해주세요" |

#### 4.4.4 계약 만료 / 해지 처리

| 항목 | 내용 |
|---|---|
| **ID** | ADM-CTR-04 |
| **우선순위** | Must |
| **설명** | 계약 상태를 EXPIRED 또는 TERMINATED로 변경한다. 상태 변경 시 해당 Space의 status를 AVAILABLE로 복원하고, 입주자의 역할을 `RESIDENT` → `USER`로 변경하여 IoT 접근 권한을 회수한다. |

**처리 흐름:**

```

5. 다음 접근 시 재로그인을 유도하여 role=USER인 새 토큰이 발급되도록 한다.
```

> [!WARNING]
> 계약 해지 시 IoT 접근 권한이 즉시 회수됩니다. 기존 토큰으로의 IoT 제어 시도는 토큰 블랙리스트 검증에 의해 거부됩니다.

## 부록: DB 역할(Role) 컬럼 설계

### USER 테이블 역할 컬럼

```sql
ALTER TABLE users
  ADD COLUMN role ENUM('USER', 'RESIDENT', 'ADMIN') NOT NULL DEFAULT 'USER';
```

| 값 | 설명 | JWT 포함 클레임 |
|---|---|---|
| `USER` | 일반 회원 (기본값) | `sub`, `role`, `exp`, `iat` |
| `RESIDENT` | 입주자 (계약 체결 완료) | `sub`, `role`, `contract_id`, `space_id`, `exp`, `iat` |
| `ADMIN` | 관리자 | `sub`, `role`, `exp`, `iat` |

### 역할 전환 이력 테이블 (선택)

```sql
CREATE TABLE role_change_log (
  id          BIGINT AUTO_INCREMENT PRIMARY KEY,
  user_id     BIGINT NOT NULL,
  old_role    ENUM('USER', 'RESIDENT', 'ADMIN') NOT NULL,
  new_role    ENUM('USER', 'RESIDENT', 'ADMIN') NOT NULL,
  reason      VARCHAR(255),           -- 예: 'CONTRACT_CREATED', 'CONTRACT_EXPIRED'
  contract_id BIGINT,                 -- 관련 계약 ID (있는 경우)
  changed_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  changed_by  BIGINT                  -- 변경 수행자 (관리자 ID 또는 시스템)
);
```
