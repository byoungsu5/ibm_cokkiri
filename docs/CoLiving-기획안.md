# 코리빙(Co-Living) IoT 플랫폼 — 기획안

**문서 목적:** 교육 과제 「프로젝트 2: 코리빙 플랫폼」의 **요구사항·범위·방법론·산출물**을 한곳에 정리하고, 팀이 **현실적인 MVP**로 합의할 수 있게 한다.

---

## 1. 요약 (Executive Summary)

### 1.1 무엇을 만드는가

**공동 거주(코리빙) 환경**에서 **IoT 기기**를 연동·관리하기 위한 **플랫폼**이다. 교과서적 표현으로는 **IoT 기기 제공 업체**와 **공동 거주 서비스(운영) 업체** 사이에서 **미들웨어(중간 플랫폼)** 역할을 하며, 운영사가 **청소기·세탁기·공용 현관·냉난방기·조명** 등 기기를 **쉽게 연결·관리**할 수 있게 한다.

**교육 과제 현실:** 실제 멀티 벤더·실제 클라우드 전체(다중 IoT)를 전부 구현하기는 어렵다. 따라서 **목업(Mock) IoT**, **단일 진입 URL(게이트웨이/BFF)**, **최소 MSA 수준의 서비스 분리**로 **데모 가능한 산출물**을 목표로 한다.

### 1.2 기술 방향 (팀 합의용 제안)

| 층                  | 제안                                                         |
| ------------------- | ------------------------------------------------------------ |
| 사용자 접점         | **Next.js** (React 계열) — 화면 + **BFF**(Route Handlers 등) |
| 도메인·DB·외부 연동 | **Spring Boot**, **헥사고날 아키텍처**, **JPA**              |
| 데이터              | **PostgreSQL**                                               |
| 목업 IoT            | HTTP JSON 서버 (WireMock / Node 등)                          |
| 실행·배포           | **Docker Compose** 중심, **CI/CD** 최소 1줄                  |

### 1.3 한 줄 결론

**"브라우저 → Next(BFF) → Spring(API) → DB + 목업 IoT"** 가 끊기지 않게 만들고, **게이트웨이 역할(라우팅·인증·로깅)** 과 **거주자·관리자 최소 화면**을 4주 안에 완주한다.

## 2. 배경 및 문제 정의

### 2.1 As-Is (가정)

- 공동 거주 운영사는 **여러 IoT 업체**의 API 스펙이 제각각이라, 거주자 앱·관리 시스템을 일일이 맞추기 어렵다.
- **단일 API 표면**과 **인증·라우팅·감사 로그**가 없으면 운영·보안이 어렵다.

### 2.2 To-Be (과제가 지향하는 상태)

- **미들웨어**가 **통합 API**를 제공하고, **게이트웨이**가 요청을 **백엔드·(목업) IoT**로 라우팅한다.
- **거주자**는 앱으로 기기를 확인·제어하고, **관리자**는 고객사·공간·기기·권한을 관리한다.

### 2.3 교육 과제에서의 제약

- **실제 IoT 디바이스 없음** → **목업 서비스** 또는 **대체 HTTP API**로 구현 (과제 권고).
- **인력·기간 대비 과제 범위가 큼**

## 3. 과제 명세와의 매핑

### 3.1 과제가 요구하는 구성 요소

| 과제 구분                | 내용                                                                                                | 기획안에서의 반영                                                                                      |

| **게이트웨이 서비스**    | API Gateway 역할, 중계, 라우팅, API 관리, 보안(인증/권한), 모니터링, 통합 API, (노코드 확장은 설계) | **BFF + (선택) Spring Cloud Gateway / Nginx** 수준으로 구현. 모니터링은 **로그·간단 대시보드(Should)** |
| **거주자**               | 거주자용, 기기 확인·제어                                                                            | **Next 거주자 UI** + **Spring 유스케이스**                                                             |
| **관리자**               | 고객사·서비스·공유 면적·기기·모니터링·시스템 권한                                                   | **관리자 UI** + **Spring admin API**. "서비스 관리"는 **기능 플래그/환경설정** 정도로 축소 가능        |
| **기술 아키텍처 (참고)** | 컨테이너·API Gateway                                                                                | **문서·다이어그램** + **Docker Compose** 로 실구현 대체                                                |

### 3.2 API Gateway 개념 (과제 설명)

- **컨슈머(클라이언트)** 는 **하나의 엔드포인트**만 알고, 뒤의 복잡한 시스템은 게이트웨이가 **라우팅·보안·로깅** 등으로 처리한다.

## 4. 범위 정의 (MoSCoW)

| M2  | **백엔드 API**: 고객사(Tenant), 공간(Unit/공유 면적), **기기 메타데이터**, 상태 조회·제어 **목업 연동** |

| M5  | **배포**: Docker Compose 기반 빌드 + **로컬 또는 학습용 서버 1곳** 배포 동작                            |
| M6  | **산출물**: 요구사항·아키텍처·ERD·API 명세·와이어프레임·테스트 결과 **주차별 누적**                     |

## 5. 기능 요구사항 (상세)

### 5.1 게이트웨이 (CLP Gateway) 및 BFF

- **라우팅**: 단일 진입점 `/api/bff/...` 를 통해 백엔드(Spring)·목업 IoT 서비스로 요청을 분배한다. 라우팅 규칙은 별도 문서로 정의한다.
- **인증 및 인가 관리**: JWT 토큰 기반 인증을 BFF 또는 백엔드 **한 곳**에서 통일하여 처리한다. 토큰 파싱·권한 검증 로직을 게이트웨이 단에 집중시켜 각 서비스의 책임을 분리한다.
- **로깅 및 모니터링**: 요청 ID(Correlation ID) 추적, **감사 로그(ControlLog)** 기록, Docker 컨테이너 상태를 확인하기 위한 **Health Check** 엔드포인트(`/actuator/health` 등) 제공.

### 5.2 시스템의 2대 역할자 (Roles)

#### ① 운영 관리자 (Operator / Tenant Admin)

- **성격:** 특정 코리빙 지점을 실제로 운영하는 **'운영사' 직원**.
- **주요 임무:**
  - 입주민 계약 관리
  - 월세/공용시설 결제 승인.
  - 공용 공간 기기 제어
  - 실시간 시설 예약 현황 관리.
  - 대시보드를 통한 지점별 에너지 사용량 및 장애 모니터링.
  - 입주민과의 1:1 문의 및 민원 처리.
- **사용 화면:** 운영 대시보드 (Admin Web), 관리자페이지

#### ② 입주자 및 게스트 (End User / Tenant User)

- **성격:** 코리빙 시설에 거주하며 서비스를 이용하는 **'실사용자'**.
- **주요 임무:**
  - 본인 방 및 공용 공간의 IoT 기기(조명, 냉난방) 원격 제어.
  - 공용 시설(세탁실, 미팅룸) 예약 및 결제.
  - 커뮤니티 게시판 이용 및 관리자에게 문의하기.
  - 본인 계약 정보 및 임대료 납부 현황 확인.
- **사용 화면:** 입주자 전용 App (Mobile)

## 7. 도메인 모델 (개념)

| 도메인 개념                | 설명                                                                                          |
| -------------------------- | --------------------------------------------------------------------------------------------- |
| **Tenant (운영사)**        | 코리빙 지점을 운영하는 회사 단위. 하나의 Tenant가 여러 Site를 보유할 수 있다                  |
| **Site (사이트)**          | 단지·건물 등 상위 물리적 지역 그룹. 하나의 Site 안에 여러 Space가 존재                        |
| **Space (스페이스)**       | 호실·방 등 실제 입주자가 거주하는 최소 공간 단위. 공용 공간(세탁실, 미팅룸 등)도 Space로 관리 |
| **Operator (운영 관리자)** | 특정 Tenant에 소속되어 계약·예약·기기·민원 등 운영 업무를 처리하는 관리자 계정                |
| **Resident (입주자)**      | 특정 Space에 배정되어, 기기 제어·시설 예약·커뮤니티 등 서비스를 이용하는 사용자               |
| **Contract (계약)**        | 입주자와 Space 간의 임대차 관계. 계약 기간·상태·요금 정보를 포함                              |
| **Device (기기)**          | Space에 설치된 IoT 기기 (조명, 냉난방 등). 종류·상태·제어 명령 정보를 가짐                    |
| **Reservation (예약)**     | 공용 시설(세탁실, 미팅룸 등) 이용을 위한 시간대별 예약 정보                                   |
| **ControlLog (제어 이력)** | 기기 제어 시 기록되는 감사 로그 (누가, 언제, 어떤 기기를, 어떤 명령)                          |
| **(외부) 목업 IoT**        | 실제 IoT를 대체하는 HTTP Mock 서버. 백엔드 아웃바운드 어댑터를 통해 통신                      |

## 9. 아키텍처

### 9.1 실구현 (Docker Compose 기반)

- **Docker Compose**: `web`(Next), `api`(Spring), `postgres`, `mock-iot`
- 모든 서비스를 **하나의 `docker-compose.yml`** 로 관리하여, `docker compose up` 한 줄로 전체 환경 기동.
- 개발 환경과 데모 환경 모두 Docker Compose를 사용하여 일관성 있는 실행 환경을 보장한다.

### 9.2 논리 구조

```
브라우저 → Next(UI) → Next(BFF) → Spring(헥사고날) → PostgreSQL
                                              ↓
                                         목업 IoT (HTTP)
```

### 9.3 디렉토리 아키텍처 (역할별 도메인 분리)

사용자의 2대 역할자(운영 관리자, 입주자)의 주요 임무를 기준으로 프론트엔드 라우팅과 백엔드 헥사고날 도메인 패키지를 시각화한 구조이다.

**1) 프론트엔드 (Next.js App Router)**
```text
frontend/src/app
│
├── 📁 (operator-admin)             # 🏢 1. [운영 관리자] - "운영 대시보드 (Admin Web)"
│   ├── 📁 contracts                # 지점 입주민의 임대차 계약 내역 관리
│   ├── 📁 billing                  # 월세 / 공용 시설 이용료 결제 승인 및 정산
│   ├── 📁 spaces                   # 공용 공간 전체 기기 제어 세팅 (타이머, 일괄 온오프 등)
│   ├── 📁 reservations             # 실시간 공용 시설 예약 현황 모니터링 및 승인/취소 관리
│   ├── 📁 dashboard                # 지점별 에너지 사용량 및 기기 장애 모니터링
│   └── 📁 voc                      # 입주민 통합 CS 채널 (1:1 채팅, 접수 문의 처리)
│
└── 📁 (resident-app)               # 📱 2. [입주자 및 게스트] - "입주자 전용 App (Mobile 지향)"
    ├── 📁 my-room                  # 본인 방 및 접근 허용된 공용 공간의 기기(조명, 냉난방) 원격 제어
    ├── 📁 facilities               # 공용 시설(세탁실, 미팅룸) 캘린더 조회, 예약 신청 및 결제
    ├── 📁 community                # 입주자 전용 커뮤니티 게시판 및 관리자 1:1 채팅 문의창
    └── 📁 my-contract              # 본인 계약 상세 정보 조회 및 당월 임대료 납부 현황 확인
```

**2) 백엔드 (Spring Boot 헥사고날 구조)**
```text
backend/src/main/java/com/coliving
│
├── 📁 tenant-operation             # 🏢 1. [운영 관리자 도메인]
```

## 10. 프로젝트 수행 방법론 (5단계)

과제에서 **워터폴 기반 5단계**를 안내하므로, **산출물·회의**를 아래에 맞춘다.

| 단계                | 목적                 | 코리빙에서 할 일                                               |
| ------------------- | -------------------- | -------------------------------------------------------------- |
| **① 준비**          | 팀·일정·WBS          | 킥오프, 역할 분담, **4주 WBS**, 도구(Git, CI)                  |
| **② 요구사항 정의** | 기능·비기능·우선순위 | **요구사항 정의서**, **MoSCoW**, Must 확정                     |
| **③ 설계**          | 구현 방법 확정       | **아키텍처**, **ERD**, **API 명세(OpenAPI)**, **와이어프레임** |
| **④ 개발**          | 구현                 | DB→API·화면 병행, **목업** 병행, 코드 리뷰·단위 테스트         |
| **⑤ 테스트**        | 품질 검증            | **통합 시나리오**, **테스트 케이스·결과서**, 비기능 최소       |

**애자일을 병행**해도 되지만, **산출물 기준**은 위 5단계로 맞추면 발표·평가에 유리하다.
